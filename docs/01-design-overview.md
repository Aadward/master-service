# 01 · 设计概览：目标、哲学与整体架构

## 1. 问题背景

ERP 系统包含多个模块（Sales、MRP、PLM、CRM、Finance …），每个模块都有自己的
customer 相关配置与表。新增一个 customer 时，需要在多达 **33 个页面**逐一配置，
存在以下痛点：

- 重复劳动：同一份核心信息要在多处填入
- 易遗漏：哪个模块漏配，常常只能事后发现
- 顺序混乱：模块间存在隐式依赖，没有统一指引
- 不可观测：找不到"这个客户卡在哪一步"的全局视图

## 2. 设计目标

| 目标 | 说明 |
|---|---|
| **统一入口** | 新增客户只在 master-service 操作一次 |
| **轻量解耦** | master-service 不替代任何域的配置职责 |
| **可观测** | 33 个配置任务的状态全程可见 |
| **可恢复** | 任意一步失败可定位、重试、补偿 |
| **可扩展** | 新增模块 / 配置页只改模板，不改核心代码 |
| **幂等** | 重复调用不产生脏数据 |

## 3. 设计哲学

> **Master-service 只对"这个客户在系统层面走到哪一步"负责，
> **不**对"这个客户在某个域里被配成什么样"负责。**

### 3.1 职责边界

| 做 | 不做 |
|---|---|
| 维护客户最小核心字段（各域交集） | 持有任何域的自定义配置内容 |
| 维护跨域配置的依赖顺序（DAG） | 校验域内字段的业务规则 |
| 广播 / 暴露"轮到你了"信号 | 主动写入其他系统 |
| 基于核心字段派生"建议配置项" | 决定下游是否采纳建议 |
| 跟踪"谁配完了 / 卡在哪一环" | 持有下游域的配置结果细节 |
| 提供面向运维的进度视图 | 做跨域的回滚 / 补偿 |

### 3.2 关键判定

- **编排 vs 编舞**：选 **编排（轻量）**。Master-service 负责"发号"，但**不发"指令"**，只发"轮到你了 + 建议"。
- **同步 vs 异步**：以 **异步事件 + 拉取兜底** 为主；同步 REST 作为旧模块兼容。
- **强一致 vs 最终一致**：**最终一致**。跨 33 个系统不可能 2PC。
- **任务粒度**：**一页（或一个域内逻辑单元） = 一个 task**。粒度细，失败定位准。

## 5. 核心组件

### 5.1 Customer MinData Registry · 最小数据字典
- 客户的唯一身份源（customer_id 全局唯一）
- 仅维护**各域交集字段**（如：name、country、industry、customer_type、currency 默认值…）
- 维护客户配置生命周期总体状态：`INIT → IN_PROGRESS → READY / PARTIAL / FAILED`

### 5.2 Template Engine · 模板引擎
- 把"33 个页面"沉淀为可维护的模板（按 customer_type 分）
- 模板声明：哪些任务、谁依赖谁、每个任务的**建议配置项派生规则**
- 模板用 YAML / JSON 描述，可热加载（详见 [04-suggested-config.md](04-suggested-config.md)）

### 5.3 DAG Coordinator · 依赖图调度器
- 客户创建时，依据模板物化所有任务（全部 WAITING）
- 维护任务依赖关系；当上游任务进入终态（DONE / SKIPPED）时，把后继 WAITING → READY
- 不主动"执行"任务，只负责"放行"

### 5.4 Notifier · 通知器
- 推（push）模式：READY 时广播事件 `customer.task.ready`
- 拉（pull）模式：暴露 `GET /customers/{id}/next-tasks?owner=...`
- 两种模式并行支持，下游按自身成熟度选择

### 5.5 State Store · 状态存储
- PostgreSQL，强事务保证状态机一致性
- 详见 [05-data-model.md](05-data-model.md)

### 5.6 Audit Log · 审计
- 记录每一次状态迁移：谁、何时、从什么状态到什么状态、原因
- 不存储下游域的配置内容快照（如需，由下游自行存）

## 6. 典型协作模式

```
1. Admin 在 UI 创建客户（只填最小集合）
2. Master 物化 33 条 task，把入度=0 的任务置 READY
3. 各下游域监听到 / 拉取到自己的 READY 任务
4. 下游：claim → 自行从 master 拿 minimum_data + suggested_config
        → 在自己系统里完成配置 → 回报 done
5. Master 移动 DAG 指针；新入度=0 的任务进入 READY
6. 直到所有 task 完成（或部分 SKIPPED）→ customer 整体 READY
```

详细时序见 [03-dag-scheduling.md](03-dag-scheduling.md)。
