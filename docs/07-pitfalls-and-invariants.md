# 07 · 易踩坑点与边界守护

> 这一份不是设计本身，而是**给未来的自己 / 团队的提醒**。
> 每一条都对应一个真实的设计偏移风险。

## 1. 不要把"建议"变成"必须"

`suggested_config` 是**只读建议**。下游覆盖了，就是下游的权威。
如果某天有人提"我们要保证下游必须用 master 派生的值" —— **这是滑向上帝服务的第一步**。

> 解药：保持派生规则为纯函数；不在 master 维护"采纳率""一致性检查"。
> 如果业务需要"中央拉齐"，那是另一个服务（数据治理）的职责。

## 2. 最小集合的字段集要克制

一旦某字段进入 minimum data，所有域都会基于它派生建议，回头改语义影响面巨大。

| 守护规则 | |
|---|---|
| 字段集只增不删 | 删除字段会让历史派生失效 |
| 字段语义变更必须升级版本号 | 不允许"原地改语义" |
| 增加字段前回答：是不是**所有域都关心**？ | 如果只有 1~2 个域关心，那不属于 minimum |

## 3. DAG 只编排"系统间顺序"，不编排"域内顺序"

域内的多步状态机（"Sales 配完资料→走二级审批→生效"）是**域自己的事**。
Master 看到的就是一个 task（或一组同模块 task），它的 done = 域内全部走完。

> 反例：在 master 模板里写"Sales 步骤1 → Sales 步骤2 → Sales 步骤3"。
> 这是越权 —— 域内顺序变了你就要改 master 模板。

## 4. 回执必须带 task_id

不要按 `(customer_id, module)` 推断 task。模板演进后同一模块可能有多个并行任务，
按 module 推断会把回执回错任务。

## 5. CLAIMED 必须设超时

例如 30 分钟未 done，自动回收到 READY，避免下游崩溃后任务被"占死"。
**没有超时回收 = 单点故障 = 整个流水线卡死**。

## 6. 失败语义必须分级

至少区分两类（`failed_reason_code`）：

| 类型 | 例子 | 处理 |
|---|---|---|
| **瞬时失败** | 下游不可用、超时、5xx | 自动指数退避重试 |
| **业务失败** | 字段校验失败、缺前置资料 | 立即升级人工，**不**自动重试 |

混淆这两类 → 要么垃圾重试浪费资源，要么业务问题被静默吞掉。

## 7. 幂等是底线

| 接口 | 幂等键 |
|---|---|
| `POST /customers` | `Idempotency-Key` + `external_ref` |
| `POST /tasks/{id}/claim` | `(task_id, owner)` |
| `POST /tasks/{id}/done` | `(task_id, owner)` |
| `POST /tasks/{id}/failed` | `(task_id, owner, retry_count)` |

下游可能因网络抖动重发，**无幂等 = 状态机被污染**。

## 8. 不要在 master-service 持有下游域的"配置内容"

哪怕只是"为了审计"，也不要。

- 审计的不变量是 **"谁在何时改了什么状态"**，不是 "配成了什么值"
- 配置值的真相在下游，master 持有就会出现**双写不一致**
- 真要审计配置值 → 下游域自己的审计日志，或者搭一个独立的"配置快照服务"

## 9. 模板版本要冻结到客户上

```sql
customer.template_version = 3
```

- 已创建的客户**不会**因为模板升级被静默改变行为
- 新客户走最新版
- 老客户的模板升级要**显式**触发（"补齐"流程，见 [08-roadmap.md](08-roadmap.md)）

## 10. WAITING / BLOCKED 不广播

只有 READY 才向下游发事件。
否则下游会被"还没轮到你"的提示淹没，对系统的信任会迅速崩坏。

## 11. SKIPPED 与 DONE 同等触发后继

把"不适用"和"完成了"在 DAG 推进上视为同一类终态。
否则下游有一个 task 不适用，就会卡住全链路。

## 12. 不要让 master-service 做跨域回滚

一旦 DONE，状态属于下游。Master 没有能力（也不应该）"把 Sales 配好的客户撤掉"。

> 如果业务真的需要"客户级回滚"，那是一个**独立的回滚编排服务** —— 不是 master 的职责。
> Master 能做的最多是把 `customer.overall_status` 置为 `CANCELLED`，并向各域**通知** "这个客户作废了，你们自行处理"。

## 13. 字典（lookup）的 owner 不一定是 master

`currency_by_country` 可能是财务团队维护，
`price_book_by_segment` 可能是销售团队维护。
Master 只是**托管 + 版本化** + 让派生规则能引用它们。

> 不要因为字典存在 master 的库里，就以为字典内容由 master 团队负责。
> 这会让 master 团队被各种业务字典维护拖死。

## 14. UI / OpenAPI 不要堆"快捷功能"

每加一个"一键批量配 100 个客户"的便利接口，背后都需要状态机能正确处理。
**保守地暴露能力**，先确保每一步可观测、可重试，再谈批量。

## 15. 监控的颗粒度要到 task

不要只监控 customer 整体状态。需要：

- 每个 task 的平均停留时长（识别"哪一步成了瓶颈"）
- 每个 module 的失败率（识别"哪个下游有问题"）
- claim 超时回收次数（识别"哪个下游不稳定"）

## 自检清单（季度回顾）

- [ ] master 数据库里有没有出现下游域才该存的字段？
- [ ] 模板里有没有出现"非纯函数"的派生规则？
- [ ] 最小集合字段数是否在膨胀？
- [ ] 有没有 task 一直停在 CLAIMED / IN_PROGRESS（超时机制是否生效）？
- [ ] 失败的 task 中，业务失败和瞬时失败是否被正确分类？
- [ ] 模板升级时，已存在客户的行为是否保持稳定？
