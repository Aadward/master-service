# 08 · 演进路径

## 阶段 1 · MVP（1~2 个月）

**目标：跑通最小闭环。**

- [ ] 数据模型：customer / config_task / config_template / audit_log
- [ ] 客户创建接口：`POST /customers`，模板物化任务
- [ ] DAG 调度器：WAITING→READY 的基本推进
- [ ] 任务接口：`claim` / `done` / `failed` / `skip`
- [ ] 模板格式定义（YAML），支持静态默认值
- [ ] 简单 Admin UI：客户列表 + 33 格状态色块
- [ ] 接入 **2~3 个核心模块**（推荐 Sales + CRM + Finance）走 REST 同步通知

**验收**：能在 UI 创建一个客户，看到 2~3 个模块依次走完。

## 阶段 2 · 完整化（3~4 个月）

**目标：覆盖全部模块、达到生产可用。**

- [ ] 引入 Kafka，加事件 `customer.task.ready` / `customer.task.failed`
- [ ] 派生规则引擎：支持 `${...}` 表达式 + 字典 lookup
- [ ] 接入剩余模块（PLM、MRP 等）
- [ ] 失败分级：瞬时 vs 业务，分别配置自动重试策略
- [ ] CLAIMED 超时回收的后台任务
- [ ] 幂等记录表 `idempotency_record`
- [ ] 监控大盘：task 停留时长、模块失败率、claim 超时次数
- [ ] OpenAPI 文档 + 下游接入 SOP

**验收**：新增客户在 0 人工干预下，33 task 平均 < 4h 全部完成。

## 阶段 3 · 治理与演进（持续）

**目标：模板可灰度、历史可回填、运维可下沉。**

- [ ] 模板版本化与灰度：新版可指定"只对 X 类型客户生效"
- [ ] **模板升级补齐**流程：
  - 新版本加了 task → 扫描已有客户 → 为缺失的 task_key 补建 WAITING 记录
  - 不修改已 DONE 的 task
- [ ] 字典版本化与灰度（同上）
- [ ] 自助式 "建议配置项 dry-run" 接口：业务方可输入虚拟 min_data，看派生结果
- [ ] 一线运维台：失败 task 重试、批量 skip、人工 block / unblock
- [ ] 模板 schema 静态校验工具（CI 时校验，避免坏模板上线）

## 阶段 4 · 可选增强

- [ ] **引入 Temporal / Camunda 重构调度逻辑**
  - 评估前提：自研调度逻辑出现频繁的死状态 / 难调试问题
  - 收益：原生的重试、补偿、状态持久化
  - 代价：引入新的运维栈，学习成本

- [ ] **跨客户批量操作**
  - 例如"把 100 个客户的 PLM 模板从 v1 升到 v2"
  - 必须先确认状态机能正确处理并发，再开放

- [ ] **配置快照服务（独立项目）**
  - 如果业务需要"中央能看到所有域的最终配置值"
  - 这是 **另一个独立服务**，不归入 master-service
  - 由各下游 push 快照到它

---

## 不做（明确划出去）

| 需求 | 由谁做 |
|---|---|
| 跨域回滚 / 撤销客户 | 独立的回滚编排服务（如有需要） |
| 配置内容的中央持久化 | 独立的配置快照服务 |
| 各域的内部审批流 | 各域自己的流程引擎 |
| 客户业务数据（订单、发票…） | 各域业务系统 |
| 客户主数据治理（去重、合并） | 独立的 MDM 服务 |

> 这张表的存在本身就是边界守护的工具：每隔几个月回看，确保还没有偷偷接管。
